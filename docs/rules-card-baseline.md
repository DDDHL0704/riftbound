## 3. 核心规则不变量

新项目必须先把这些不变量建成测试，再实现复杂卡牌：

- 卡面文字优先于规则；PDF 不覆盖或冲突时按官网卡面。
- “无法”优先级高于“可以”。
- 结算时尽可能执行所有指示，无法实现的指示忽略；所有指示无法实现时仍可打出并结算为空效果。
- 卡牌不会进入其他玩家所属的非场地区域；进入时改为进入所属者对应区域。
- 所有权与控制权分离：`ownerId`、`controllerId` 必须贯穿单位、装备、战场控制、目标、区域移动、得分、触发和替换。
- 隐秘/私密/公开信息必须由服务端视角快照裁决，前端不能拿到不该看的信息。
- 打出卡牌与激活技能必须走统一步骤：移到结算链、选择、确定总费用、支付、合法性检查、确认、结算。
- 法术对决、优先行动权、焦点、让过和结算链必须可由事件流回放。
- 触发式技能、替换效果、清理任务必须有明确队列和排序策略。
- 战斗、移动、召回和得分是核心闭环，不可做成卡牌特例。
- 贴附、顶部卡牌、未激活文本和装备控制权是装备系统的核心，不可只用宿主字段模拟。
- 经验、增益、额外伤害、效果层、等级依赖必须独立建模，不能散落在卡牌脚本里。

## 5. 卡牌实现原则

卡牌开发按功能逻辑单元推进，而不是按 1009 个官方条目逐张硬写：

- 图鉴、收藏、卡面展示保留每个官方条目。
- 后端规则按 811 个功能逻辑单元去重。
- 异画、复刻、同名同效果卡共享一个规则实现。
- 单张牌如果有多个独立分支，按多个行为模板和测试点管理。
- 所有规则文本必须保留官方原文、结构化解析结果、实现状态和测试证据。

卡牌数据分层：

| 层 | 内容 | 来源 |
|---|---|---|
| OfficialCard | 官网原始条目、编号、图片、名称、类型、颜色、费用、规则文本 | 官网快照 |
| FunctionalUnit | 行为等价分组、代表编号、所有映射条目 | 生成工具 |
| CardDefinition | 游戏内可实例化定义、基础数值、标签、关键词 | 新项目结构化数据 |
| BehaviorSpec | 解析后的行为模板、触发条件、费用、目标、效果 | 新项目规则数据 |
| RuleImplementation | C# 规则执行器、触发器、替换器、动作实现 | 新项目代码 |
| Evidence | PDF 规则点、官网文本、oracle fixture、conformance 测试 | 测试与文档 |

## 6. 当前旧实现参考价值

旧 Java 项目有三类价值：

- 已验证规则行为：房间串行、幂等、事件广播、玩家视角、基础战斗、支付、装备、控制权。
- oracle fixture 来源：当前 JUnit 和 Room 测试能导出 command log、events、snapshots。
- 覆盖矩阵参考：`docs/card-effect-buckets.generated.md`、`docs/card-functional-equivalence.generated.md`、`docs/card-effect-test-matrix.generated.md`。

但新项目不照搬这些边界：

- 不照搬 Spring WebSocket 层，改为 ASP.NET Core + SignalR。
- 不照搬可变对象散落式规则，逐步收敛到 command -> events -> snapshots。
- 不照搬前端调试界面，后续会基于新协议设计产品级 Web 对战界面。

## 7. 第一批必须覆盖的规则领域

在写大批卡牌前，必须先完成这些规则领域：

1. 游戏对象、区域、隐私、所有权和控制权。
2. 玩家、房间、座位、重连、命令幂等。
3. 主牌堆、符文牌堆、手牌、废牌堆、放逐区、传奇区、英雄区、基地、战场、待命区。
4. 回合、阶段、开环/闭环、法术对决、焦点、优先行动权。
5. 打出卡牌、激活技能、费用确定、费用支付、合法性检查。
6. 结算链、触发队列、替换队列、清理队列。
7. 移动、召回、待命、活跃/休眠、眩晕、增益、伤害、摧毁、放逐、回收、抽牌。
8. 战斗、伤害分配、壁垒/后排、得分、胜负。
9. 装备、贴附、卸除、顶部卡牌、未激活文本。
10. 经验、等级、百炼、装配、灵便、回响、伏击等高频关键词。

## 8. 基线更新规则

以后每次官网快照、规则 PDF 或生成矩阵更新，必须同步：

- 重新生成卡牌统计与功能等价分组。
- 标注规则或卡面冲突。
- 更新主开发计划中的阶段范围和验收清单。
- 对已实现功能运行 conformance tests，确认没有被数据更新破坏。
