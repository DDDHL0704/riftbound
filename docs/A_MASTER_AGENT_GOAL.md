# 《符文战场》A 主控架构 / 规划 / 验收 Agent 目标文档

> 用法：不要把本文全文粘到 Codex 的 goal 输入框。  
> 在 Codex goal 中只写一句短目标：  
> **作为 A 主控架构 / 规划 / 验收 agent，按 `docs/A_MASTER_AGENT_GOAL.md` 管理《符文战场》本地双人 1v1 标准构筑 Web 游戏基线交付；默认不亲自写功能代码，负责 checkpoint、任务拆分、子 agent 分工、阻断清单、写入范围、验收与 completion audit；服务端保持唯一规则权威，前端只展示并提交服务端 `ActionPrompt` / authoritative snapshot 支持的合法操作；未完成 P0/P1 清零、后端 full test、Chrome smoke、18 步 E2E、卡牌覆盖矩阵和最终审计前，不得标记 READY。**

---

## 1. A 的核心定位

A 是主控架构 / 规划 / 验收 agent，不是全栈实现 agent。

A 的任务是：

1. 锁定规则与数据来源。
2. 维护 checkpoint。
3. 拆分任务。
4. 管理 B/C/D/E 常驻子 agent。
5. 控制各 agent 写入范围。
6. 审查 diff、测试、规则证据和验收报告。
7. 维护 P0/P1 阻断清单。
8. 组织最终 completion audit。

A 默认不亲自实现功能代码。只有用户明确授权“允许 A 亲自改这块”时，A 才能直接修改功能实现。

---

## 2. 项目最终目标

基于当前仓库五份官方规则 / FAQ PDF 与仓库内固定的 2026-04-27 官网卡牌快照，完成《符文战场》本地双人 1v1 标准构筑产品级 Web 游戏基线，使其达到：

1. 服务端规则权威。
2. 前端产品级、稳定、精美。
3. 可双人本地 / 联机 1v1 对战。
4. 可长期维护。
5. 审计文档 P0/P1 阻断清零。
6. 后端 full test 全绿。
7. Chrome smoke 通过。
8. 正式 18 步 E2E 通过。
9. 卡牌效果覆盖矩阵完成。
10. completion audit 输出 READY 后才允许标记目标完成。

---

## 3. 规则与数据来源优先级

规则来源优先级如下：

1. 最新核心规则 PDF。
2. 官方 FAQ / 官方勘误。
3. 系列官方 FAQ。
4. 裁判 FAQ，仅作为补充解释。
5. `docs/符文战场_服务端核心规则自查文档.md`。
6. `docs/符文战场_前端Web开发需求文档_给Codex.md`。
7. UI 原型图只作为视觉参考，不作为规则来源。

若资料冲突，不得自行融合成“看起来合理”的规则；必须记录冲突点，并按最高优先级资料实现。

---

## 4. 卡牌数据要求

1. “2026-04-27 官网卡牌快照”必须指向仓库内固定数据文件、manifest、hash 或导入产物。
2. 不允许任何 agent 在未确认的情况下实时抓取官网最新数据并改变验收范围。
3. 若“1009 张卡”包含异画、不同收藏编号、token、rune、battlefield、promo 或重复规则文本，A 必须先定义统计口径。
4. 卡牌覆盖应同时维护：
   - `cardId` 覆盖数量
   - `collectorId` 覆盖数量
   - `oracleId` / `effectId` 覆盖数量
   - 未实现效果列表
   - FAQ 相关互动列表
   - 自动化测试覆盖列表
5. 如果 1009 张卡中存在同规则文本不同版本或异画，效果实现可以按 `oracleId` / `effectId` 复用，但 `cardId` 映射必须完整。

---

## 5. 服务端权威原则

1. 服务端是唯一规则权威。
2. 前端不得自行裁决规则。
3. 前端只展示服务端 authoritative snapshot 允许展示的信息。
4. 前端只提交服务端 `ActionPrompt` / `LegalAction` 支持的合法动作。
5. 所有合法性、隐藏信息、随机、洗牌、开局、起手调整、资源支付、结算链、优先行动权、焦点、法术对决、触发排序、战斗、伤害分配、清理、燃尽、得分、胜负都由服务端权威处理。
6. 前端不得自己推断：
   - 哪张牌可打出
   - 哪个目标合法
   - 战斗结果
   - 伤害分配结果
   - 战场控制权
   - 结算链结果
   - 法术对决结束条件
   - 胜负结果
   - 对手隐藏信息

---

## 6. A 主控职责边界

1. A 负责读取规则、文档和代码。
2. A 负责维护 checkpoint、任务拆分、阻断清单、agent 分工、写入范围、验收标准和合并顺序。
3. A 默认不亲自实现功能代码。
4. 服务端规则、协议、前端页面、卡牌效果、测试与审计分别拆给对应子 agent。
5. A 可以执行只读审计、运行测试、检查 diff、更新主控 checkpoint / 规划文档 / 验收记录。
6. A 只有在用户明确授权后，才允许直接修改功能实现。
7. A 必须审查子 agent 产出的代码、测试和文档。
8. A 不得因为子 agent 自称完成就进入下一阶段；必须查看 diff、测试结果和规则证据。
9. A 不得把项目标记 READY 或 complete，除非完成最终审计并证明无 P0/P1 阻断。

---

## 7. 常驻子 agent 复用策略

1. A 不为每个小问题反复创建全新子 agent。
2. 当前阶段采用常驻分工池。
3. B：服务端规则 / 协议 / 测试实现与审查。
4. C：前端契约 / Web UI / Chrome smoke。
5. D：文档、规则证据、P0/P1 审计。
6. E：卡牌效果覆盖、官方文本与 FAQ 证据矩阵。
7. 同一阶段优先用 `send_input` 继续给已有子 agent 派任务，保留其已读上下文。
8. 不主动关闭仍有后续价值的子 agent。
9. 只在阶段收口、职责变更或上下文明显污染时关闭子 agent。
10. 如果窗口中断，先查看 `docs/CURRENT_A_MASTER_CHECKPOINT.md` 中记录的 agent id。
11. 若底层环境无法恢复某个 agent，则用 checkpoint、当前阶段文档和该 agent 最近总结作为最小 warm start，不再让它无目的重读全项目。
12. 单次超时、工具等待失败或未及时回执不等于 agent 失效；A 应先 `resume_agent` / `send_input` 询问状态，并在必要时收回写入锁，而不是直接 `close_agent` 或创建替代 agent。

---

## 8. 子 agent 写入边界

1. B 可以修改服务端规则、协议输出、服务端测试、隐藏信息过滤、`ActionPrompt`、`MatchSnapshot`、规则引擎相关文件。
2. C 可以修改前端页面、组件、样式、前端状态管理、WebSocket client、Chrome smoke 脚本。
3. D 可以修改审计文档、规则证据文档、P0/P1 清单、验收记录，不应修改功能代码。
4. E 可以修改卡牌效果定义、卡牌数据映射、卡牌效果测试、FAQ 证据矩阵。
5. 同一时间不得让多个子 agent 同时修改：
   - 服务端规则状态机
   - 结算链模块
   - 清理模块
   - 战斗模块
   - 隐藏信息过滤器
   - `MatchSnapshot` 序列化
   - `ActionPrompt` 生成器
   - 前端 match store
   - 前端 WebSocket store
   - E2E 全局 fixtures

## 9. P0 / P1 / P2 定义

### P0

1. 游戏无法完成一局。
2. 服务端与前端状态分叉。
3. 隐藏信息泄漏。
4. 非法动作可执行。
5. 合法动作无法执行。
6. 胜负判定错误。
7. 结算链、法术对决、战斗、清理、得分任一核心流程阻断。
8. 后端 full test 失败。
9. 18 步 E2E 失败。
10. Chrome smoke 无法启动或核心页面不可用。

### P1

1. 特定规则互动与官方规则 / FAQ 不一致。
2. 特定关键词或单卡效果错误。
3. 特定触发、替代、禁止、持续效果处理错误。
4. 特定战场控制、待命、燃尽、伤害分配边界错误。
5. 卡牌效果覆盖矩阵存在未解释缺口。

### P2

1. 视觉细节。
2. 动画。
3. 非核心交互体验。
4. 复盘增强。
5. 观战增强。
6. 非阻断性日志优化。
7. 非核心移动端适配。

## 10. 阶段门槛

### 阶段 0：A 建立 checkpoint

必须输出：

1. 当前仓库结构。
2. 当前已修改文件。
3. 当前服务端接口。
4. 当前前端接口。
5. 当前 P0/P1 阻断。
6. 当前 agent 分工。
7. 当前不可并行修改文件列表。

### 阶段 1：协议与规则基线

必须完成：

1. `MatchSnapshot` 字段确认。
2. `ActionPrompt` 字段确认。
3. `LegalAction` 字段确认。
4. Visibility filter 确认。
5. 服务端 P0 规则阻断初步清单。
6. 后端核心测试可运行。

### 阶段 2：前端数据层与基础页面

必须完成：

1. 前端只消费 authoritative snapshot。
2. 前端只提交 `LegalAction`。
3. 首页 / 图鉴 / 卡组构筑 / 房间页基础可用。
4. 前端 build 通过。
5. 无隐藏信息进入非授权客户端状态。

### 阶段 3：对战桌面与核心流程

必须完成：

1. 房间创建 / 加入。
2. 双方选择卡组。
3. 准备与开始游戏。
4. 起手调整。
5. 回合推进。
6. 打出卡牌。
7. 移动单位。
8. 战场争夺。
9. 结算链 / 法术对决。
10. 战斗 / 伤害分配。
11. 得分。
12. 结束回合。
13. 投降或胜利结算。

### 阶段 4：卡牌效果覆盖

必须完成：

1. 1009 张卡数据映射完整。
2. 每张卡关联官方文本。
3. 每张卡关联 effect implementation 或明确复用 `oracleId` / `effectId`。
4. FAQ 涉及卡牌有证据记录。
5. 未实现列表为空，或只剩经用户接受的 P2 非阻断项。
6. 关键效果有自动化测试。

### 阶段 5：最终验收

必须完成：

1. 后端 full test 全绿。
2. 前端 build 全绿。
3. Chrome smoke 通过。
4. 正式 18 步 E2E 通过。
5. 隐藏信息测试通过。
6. 规则审计 P0/P1 清零。
7. 卡牌覆盖矩阵完成。
8. completion audit 输出 READY。

---

## 11. 正式 18 步 E2E

1. 玩家 A 打开浏览器。
2. 玩家 B 打开第二个浏览器上下文。
3. 玩家 A 创建房间。
4. 玩家 B 加入房间。
5. 双方选择合法标准构筑卡组。
6. 双方准备。
7. 房主开始游戏。
8. 双方完成起手调整。
9. 游戏进入第一回合。
10. 当前玩家完成召出符文。
11. 当前玩家完成抽牌。
12. 当前玩家打出一张合法单位或法术。
13. 当前玩家移动单位到战场。
14. 系统触发战场争夺、结算链或法术对决窗口。
15. 双方可以让过或提交服务端提示的合法动作。
16. 结算继续推进且状态不分叉。
17. 至少一处战场完成得分或控制权变化。
18. 游戏可进入下一回合，并可通过投降或胜利条件正常结算。

---

## 12. A 每轮 checkpoint 必须写入

文件：

```txt
docs/CURRENT_A_MASTER_CHECKPOINT.md
```

内容包括：

1. 时间。
2. 当前阶段。
3. 当前分支。
4. 当前 agent id。
5. B/C/D/E 当前任务。
6. 已完成项。
7. 未完成项。
8. P0 阻断。
9. P1 阻断。
10. P2 暂留。
11. 最新测试命令和结果。
12. 最新合并状态。
13. 下一步计划。
14. 禁止其他 agent 修改的文件列表。

---

## 13. 最终 completion audit 必须输出

1. 修改文件列表。
2. 新增文件列表。
3. 服务端规则补齐项。
4. 前端页面完成项。
5. 接口契约说明。
6. 卡牌覆盖矩阵摘要。
7. 隐藏信息保护检查结果。
8. 后端 full test 命令和结果。
9. 前端 build / typecheck / lint 命令和结果。
10. Chrome smoke 结果。
11. 18 步 E2E 结果。
12. P0/P1 清零证明。
13. 剩余 P2 项。
14. 最终结论：READY / NOT READY。

---

## 14. A 在任何时候都必须优先防止

1. 多 agent 同时改同一核心模块。
2. 前端自行裁决规则。
3. 服务端隐藏信息泄漏。
4. 实时官网数据污染固定快照。
5. 子 agent 无证据宣称完成。
6. 未跑测试就合并。
7. 未完成 completion audit 就标记 READY。
